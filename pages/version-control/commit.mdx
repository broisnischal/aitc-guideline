# Git Commit Message Guide

## Introduction

Writing clear and meaningful commit messages is essential for maintaining a well-documented and organized codebase. Follow these guidelines to create commit messages that convey context and help collaborators understand the changes.

## Commit Structure

A commit message generally consists of three parts: type, optional scope, and a concise summary.

### Type

- **feat:** A new feature for the user.
- **fix:** A bug fix.
- **chore:** Routine tasks, maintenance, or tooling changes.
- **docs:** Documentation changes.
- **style:** Code style changes (formatting, indentation).
- **refactor:** Code refactoring without adding new features or fixing bugs.
- **test:** Adding or modifying tests.

### Scope (Optional)

Scope defines the module, component, or section of the project affected by the change.

### Summary

A brief and clear description of the changes made in the commit.

## Examples

### Example 1: Adding a new feature

```markdown
feat(user-auth): implement OAuth2 login functionality
```


### Example 2: Fixing a bug

```markdown
fix(api): resolve issue with incorrect response format
```

### Example 3: Routine maintenance

```markdown
chore(build): update dependencies to the latest versions
```

### Example 4: Documentation update
```
docs(readme): add installation instructions
```

### Example 5: Code style changes

```markdown
style(eslint): enforce consistent code formatting
```

### Example 6: Code refactoring

```markdown
refactor(user-service): improve code efficiency in user service
```
